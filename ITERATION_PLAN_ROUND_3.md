## 第1阶段：全面审计（当前 Round 0）

### 产品设计审查
**审查内容：**
1. `SPEC.md` 与实际代码实现的差异
2. API 端点完整性（SPEC 承诺的 vs 实际注册到 main.py 的）
3. 功能遗漏清单
4. 非功能需求差距（性能、安全、高可用）
5. 前端功能完整性

**交付物：**
- `AUDIT_PRODUCT.md` - 产品设计审计报告，列出每个问题及严重级别（P0/P1/P2）

### 代码质量审查
**审查内容：**
1. main.py 未注册的 router（enhanced_router、monitoring_router）
2. main.py 未注册的 service import
3. routers/__init__.py 缺失的导出
4. 新增 py 文件的 import 链完整性
5. 硬编码、超大函数、重复代码
6. 错误处理不统一的地方

**交付物：**
- `AUDIT_CODE.md` - 代码质量审计报告

### 测试完整性审查
**审查内容：**
1. 单元测试覆盖缺口（当前33个测试覆盖了哪些模块，哪些没覆盖）
2. `integration_test.py` 和 `load_test.py` 不是 pytest 测试文件（没有 `test_` 前缀函数）
3. 验证测试是否能真正运行
4. 补充关键路径测试

**交付物：**
- `AUDIT_TEST.md` - 测试审计报告

## 第2阶段：修复优先级排序

根据审计结果，将所有问题分为：

**P0（拦截级）- 必须修复才能继续迭代：**
- main.py 缺失 router 注册导致功能不可用
- 核心同步链路上的 bug
- 测试框架完全不能用

**P1（严重级）- 影响企业交付质量：**
- API 端点缺失或不完整
- 错误处理不完善
- 安全漏洞

**P2（改进级）- 体验优化：**
- 代码风格
- 文档补充
- 小的 UX 改进

## 第3阶段：开发修复（每个 P0/P1 问题走一轮）

### 标准流程：
```
问题描述 → 开发自检 → 修复代码 → 提交 → 测试验证 → QA review → 通过/打回
```

### 打回标准：
- 功能不正确实现
- 引入新 bug
- 测试未覆盖修复路径
- 代码风格不合规
